Why should you animate the UI?

  1. Animations make apps feel faster
  2. Animating between two states helps users understand the flow
  3. They inform and shape attention
  4. Reduces effort to understand which reduces cognitive load

Just as subtle Typography can change the context and our mood so can Motion Design.

Even large organisations like IBM have recognised this and made animations part of their design guidelines.

It is typical to think of animations as "Cool", "Delightful", etc.  But this should be in no way the main purpose of transitional interfaces (animations between screens, pages and states). Animations should be about improving usability.

Web Animations

The web has been fraught with challenges, with rendering inconsistencies and differences in devices and circumstances of the user.

Typically to achieve complex & choreographed animations you give up control to a framework or learn how a heavyweight library works.  However, these closed ecosystems are not really in the spirit of the web. To add insult to injury the frameworks often treat animations as a second class citizen and the libraries have unusual conventions beyond normal HTML/CSS or questionable commercial influences.

Thankfully the upcoming Web Animations API is a huge step forward,  equipped with the polyfill we can start taking advantage of it.

Web Animations API specification (or WAAPI), […] defines a model for synchronization and timing of changes to the presentation of a Web page.

Choreography.js

With classes and transitions we have learned to embrace JavaScript as a great way to trigger states and coordinate with CSS-based rendering.

This library comes about as a lightweight way to manage animations. As your actually using standardised APIs to create your animations, your not locked in and can take advantage of the wider web community.

Hackable

Animations shouldn't be just simply a ‘fire and forget’ thing. They ought to respond to the context and conditions of the user or device.

Thus, Choreography is intended to be hackable and allow for exploration of ground breaking innovations for the web.

One interesting examples is on this website.  When the device battery level is below 20%  all Choreography animations are disabled globally.

Second hackable idea I have implemented is the following:

  • Every state transition increments a counter
  • The next time the transition is played: the animation is run x% faster

Why?  If you have viewed an animation for the 100th time,  I'm sure you'd understand it quite well already. Therefore, speed it up!

Our user is a human,  they learn and adapt,  our animations should respect their time.

Got any other hackable ideas to try? Please submit them!

FAQ

Will this be Open Source?

Hell yes.
With MIT License.

How do I get it?

Still working out the kinks & API.  So no stable version has been defined yet.
The GitHub repo is open to the public: github.com/Martin-Pitt/Choreography

Can I use this in production?

This library is still in early development and not ready for production apps yet.
Natively the Web Animations API does not have much browser support currently:
caniuse.com/#feat=web-animation
There is a polyfill available though, which does seem to have effective cross-brower support: github.com/web-animations/web-animations-js
I recommend using the web-animations-next build.
